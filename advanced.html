4. High Load System (Performance Tuning - Advanced)

When a system is under high load, identify whether the issue is CPU, Memory, Disk I/O, or Network.

# Load & CPU
uptime
top
htop
mpstat -P ALL 1

# Process-level analysis
ps -eo pid,ppid,cmd,%mem,%cpu --sort=-%cpu | head
pidstat 1

# Memory analysis
free -m
vmstat 1
sar -r 1 5

# Disk I/O
iostat -x 1
iotop
df -h

# Network
netstat -i
ss -s
sar -n DEV 1 5

# System-wide tracing
dstat
atop

Step-by-Step Troubleshooting:

1. Check Load Average
   uptime

2. Identify CPU bottleneck
   top / mpstat

3. Identify Memory issue
   free -m / vmstat

4. Check Disk I/O wait
   iostat -x 1 (look for %wa)

5. Check Network saturation
   sar -n DEV

6. Drill down to process level
   ps / pidstat

7. Take action (kill / optimize / scale)

Common Root Causes:

  • High CPU → Infinite loop / bad code / heavy traffic
  • High Memory → Memory leak
  • High I/O wait → Slow disk / DB overload
  • High Load Avg but low CPU → I/O bottleneck

Fix Strategies:

  • Restart or kill problematic process
  • Optimize application queries/code
  • Scale horizontally (add servers / pods)
  • Upgrade hardware (CPU / RAM / SSD)
  • Use caching (Redis, CDN)

Interview Scenario:

“Server load is 20 but CPU is idle” → Likely Disk I/O issue → Check iostat → High wait → Move to SSD or optimize DB queries.

5. Disk I/O Bottleneck (Advanced Troubleshooting)

Disk I/O bottlenecks occur when read/write operations are slow, causing high latency and system slowdown.

Step 1: Identify I/O Wait

top        # Check %wa (I/O wait)
vmstat 1   # Check 'wa' column
uptime     # High load with low CPU → likely I/O issue

Step 2: Analyze Disk Performance

iostat -x 1

# Key metrics:
# %util → If near 100%, disk is fully utilized
# await → High value means slow disk response
# svctm → Service time per request

Step 3: Identify High I/O Processes

iotop
pidstat -d 1
lsof | grep deleted

Step 4: Check Disk Usage

df -h
du -sh /* | sort -h

Step 5: Check Filesystem Health

dmesg | grep -i error
smartctl -a /dev/sda

Step 6: Check Mount Options

mount | column -t
cat /etc/fstab

Advanced Debugging Tools:

# Real-time disk stats
dstat -dny

# Per-process I/O
atop

# Block-level tracing
blktrace /dev/sda

Common Root Causes:

  • Heavy database queries
  • Too many read/write operations
  • Slow disks (HDD vs SSD)
  • Log files growing continuously
  • Swap usage due to low memory

Fix Strategies:

  • Move to SSD/NVMe storage
  • Optimize database queries (indexing)
  • Enable caching (Redis, Memcached)
  • Rotate and clean logs
  • Increase RAM to reduce swap usage
  • Separate disks for OS, logs, and DB

Scenario 1: High Load but CPU Idle

uptime
iostat -x 1

Observation: %util = 100%, await high → Disk bottleneck

Fix: Move workload to faster disk or optimize queries

Scenario 2: Application Slow (DB Issue)

iotop
mysql slow query log

Fix: Optimize queries, add indexes

Scenario 3: Deleted Files Still Consuming Space

lsof | grep deleted

Fix: Restart process holding file

Scenario 4: Swap Causing Disk I/O

free -m
vmstat 1

Fix: Increase RAM or reduce memory usage

Scenario 5: Log File Explosion

du -sh /var/log/*

Fix: Configure logrotate

Interview Scenario:

“System is slow, CPU is idle, load is high” → Check iostat → High await → Disk issue → Optimize DB or upgrade disk.

6. Docker Image Optimization (Advanced)

Optimizing Docker images reduces size, improves security, and speeds up deployment.

# Check image size
docker images

# Analyze layers
docker history image_name

# Remove unused images
docker image prune -a

# Remove dangling images
docker image prune

# Remove unused containers, volumes, networks
docker system prune -a

# Inspect image details
docker inspect image_name

Multi-Stage Build Example:

# Stage 1 - Build
FROM node:18 AS builder
WORKDIR /app
COPY . .
RUN npm install && npm run build

# Stage 2 - Production
FROM nginx:alpine
COPY --from=builder /app/build /usr/share/nginx/html

Layer Optimization:

# Bad (creates multiple layers)
RUN apt update
RUN apt install -y nginx

# Good (single layer)
RUN apt update && apt install -y nginx && rm -rf /var/lib/apt/lists/*

Use .dockerignore:

node_modules
.git
*.log

Best Practices:

  • Use minimal base images (alpine, distroless)
  • Avoid installing unnecessary packages
  • Combine RUN commands to reduce layers
  • Use multi-stage builds for smaller images
  • Do not store secrets inside images
  • Run containers as non-root user

Security Best Practice:

# Create non-root user
RUN useradd -m appuser
USER appuser

Troubleshooting Scenario 1: Large Image Size

docker images
docker history image_name

Fix: Remove unnecessary files, use alpine, multi-stage build

Troubleshooting Scenario 2: Slow Build Time

# Use caching effectively
COPY package.json .
RUN npm install
COPY . .

Fix: Optimize Dockerfile order to leverage cache

Troubleshooting Scenario 3: Container Vulnerabilities

docker scan image_name

Fix: Update base image, remove unused packages

Troubleshooting Scenario 4: Container Running as Root

docker exec -it container_id id

Fix: Use USER directive

Interview Scenario:

“Docker image size is 1GB” → Check layers → Remove unnecessary packages → Use multi-stage → Reduce to ~100MB.

8. Security Incident (Unauthorized Access - Advanced)

This scenario covers detecting, investigating, and responding to unauthorized access on a Linux server.

Step 1: Check Login Activity

last
lastb
who
w

Step 2: Check Authentication Logs

cat /var/log/secure
grep "Failed password" /var/log/secure
grep "Accepted password" /var/log/secure
journalctl -u sshd

Step 3: Identify Suspicious IPs

grep "Failed password" /var/log/secure | awk '{print $11}' | sort | uniq -c | sort -nr

Step 4: Check Running Sessions

who
w
ps -ef --forest

Step 5: Check Open Ports & Connections

ss -tulnp
netstat -antp
lsof -i

Step 6: Check New or Suspicious Users

cat /etc/passwd
cat /etc/shadow
lastlog

Step 7: Check File Changes (Backdoors)

find / -mtime -1
find / -perm -4000

Step 8: Check Cron Jobs (Persistence)

crontab -l
ls -la /etc/cron*

Step 9: Immediate Containment

# Block attacker IP
firewall-cmd --add-rich-rule='rule family=\"ipv4\" source address=\"ATTACKER_IP\" reject' --permanent
firewall-cmd --reload

# Kill suspicious session
pkill -u username

# Disable compromised user
usermod -L username

Step 10: Secure SSH

vi /etc/ssh/sshd_config

# Recommended changes:
PermitRootLogin no
PasswordAuthentication no
AllowUsers trusted_user

systemctl restart sshd

Advanced Security Checks:

# Check listening services
ss -plnt

# Check hidden processes
ps auxf

# Check system integrity (RPM-based)
rpm -Va

# Check for rootkits
chkrootkit
rkhunter --check

Recovery & Hardening:

  • Rotate all passwords and SSH keys
  • Apply security patches (yum update)
  • Enable firewall and restrict ports
  • Install Fail2Ban to block brute force attacks
  • Enable audit logging (auditd)

Fail2Ban Setup Example:

yum install fail2ban
systemctl start fail2ban
systemctl enable fail2ban

Real-World Scenario:

Multiple failed SSH attempts detected → Identify attacker IP → Block via firewall → Disable password login → Enable SSH key authentication → Install Fail2Ban.

Golden Incident Response Flow:

1. Detect (logs, alerts)
2. Analyze (who, when, how)
3. Contain (block IP, disable user)
4. Eradicate (remove malware/backdoor)
5. Recover (restore system)
6. Harden (prevent future attacks)

11. Load Balancer Issue (Advanced Troubleshooting)

Load balancers distribute traffic across multiple servers. Issues can cause downtime, uneven traffic, or slow response.

Step 1: Check Load Balancer Reachability

curl -I http://lb-ip
ping lb-ip
traceroute lb-ip

Step 2: Check Backend Server Health

curl http://backend-ip:port
systemctl status nginx

Step 3: Verify Load Balancer Configuration

# Nginx LB config example
upstream backend {
    server 10.0.0.1:80;
    server 10.0.0.2:80;
}

# Reload config
nginx -t
systemctl reload nginx

Step 4: Check Open Ports & Connectivity

ss -tulnp
telnet backend-ip 80
nc -zv backend-ip 80

Step 5: Check Logs

tail -f /var/log/nginx/access.log
tail -f /var/log/nginx/error.log

Step 6: Check Load Distribution

# Check which backend is responding
curl http://lb-ip

Step 7: Check Health Checks

If health checks fail, LB will stop sending traffic.

# Example: check endpoint
curl http://backend-ip/health

Step 8: DNS Issues

nslookup domain.com
dig domain.com

Step 9: SSL/TLS Issues

openssl s_client -connect domain.com:443

Advanced Debugging:

# Check connections
netstat -ant | grep :80

# Monitor traffic
tcpdump -i eth0 port 80

# Check system limits
ulimit -n

Common Root Causes:

  • Backend server down
  • Health check failure
  • Wrong configuration
  • Port/firewall blocked
  • SSL misconfiguration
  • Uneven load distribution

Fix Strategies:

  • Restart backend services
  • Fix health check endpoints
  • Correct LB configuration
  • Open required ports
  • Renew SSL certificates
  • Enable sticky sessions if needed

Scenario 1: Website Accessible via Backend but Not LB

curl backend-ip
curl lb-ip

Root Cause: LB misconfiguration or health check failure

Scenario 2: One Server Getting All Traffic

Root Cause: Load balancing algorithm issue or unhealthy nodes

Scenario 3: 502 Bad Gateway

tail -f /var/log/nginx/error.log

Root Cause: Backend down or wrong upstream config

Scenario 4: Slow Response via LB

iostat -x 1
top

Root Cause: Backend performance issue

Scenario 5: SSL Handshake Failure

openssl s_client -connect domain.com:443

Root Cause: Expired or misconfigured certificate

Kubernetes Load Balancer Debugging:

kubectl get svc
kubectl describe svc service-name
kubectl get endpoints
kubectl get ingress

Golden Troubleshooting Flow:

1. Check LB accessibility
2. Check backend health
3. Verify config
4. Check logs
5. Check network/DNS/SSL
6. Fix & validate

Interview Scenario:

“Users report intermittent failures” → Check LB logs → Identify unhealthy backend → Remove node → Restore service.
